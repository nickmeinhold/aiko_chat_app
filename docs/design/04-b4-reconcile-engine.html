Design 04 — B4 reconcile engine

The orchestration policy that turns the cache's atomic primitives into a message that survives send → disconnect → reconnect → ack with no duplicate, no loss, and a correct delivery state at every step.

Round 2 (cage-match revisions). Round 1 (Maxwell/Kelvin/Carnot) returned three REQUEST_CHANGES that rhymed on one principle: the design named a mechanism (re-entrancy guard, orphan backfill, systemic-error policy, ack-wait, history delta) and gestured at it instead of specifying the lock / termination / taxonomy. Round 2 specifies each one concretely or explicitly degrades it, and adds the one race the author under-told: self-echo — the gateway fans a message back to its own sender (hub.fanout has no sender exclusion, verified hub.py:44), so the collapse path fires on every send, not just reconnect.
Component scope (Phase 1, text-only): the ChatRepository — the single writer to the drift cache (Component 3) and the only component that wires the ChatTransport (Component 1) and ChatRestApi (Component 2) together. It owns policy: when to drain the outbox, how to order reconnect, which DeliveryState transition fires when. It owns no storage and no invariant enforcement — those live in the cache, and B4 navigates them rather than re-implementing them. The line is the whole point: because the cache makes "one row per logical message" true at the write boundary, B4 cannot duplicate even with a buggy ordering — the worst a B4 bug can do is mis-order or stall, never corrupt.

Naming: "B4" is the plan's label (~/.claude/plans/bright-honking-honey.md §B4); the shipped artifact is lib/features/chat/data/chat_repository.dart. "Reconcile engine" = the policy layer; the cache is the mechanism it drives.

Who this ships to — the connectivity reality that shapes every choice

This is not a desktop-on-stable-wifi chat app. It ships to the AMR (AI, ML & Robots) crew — Andy Gelme's Melbourne meetup, whose stack is aiko_services over ESP32 / MicroPython, LeRobot SO-101 arms, and live demos on library wifi. The domain model already encodes this: SenderKind.robot is a first-class participant, and the requested @@robotname addressing means robots are chat peers, not bystanders.

Design consequence: a peer dropping mid-send and rejoining is the main case, not an edge case. So B4's reconnect choreography is the load-bearing surface, and every "await" on the network path must be best-effort with a timeout — a robot that never returns an ack must never wedge the reconnect. The correctness budget is spent on the disconnect/reconnect path because that is where the users actually live.

The contract B4 inherits — verified, not assumed

Two upstream contracts make B4 tractable. Both were read from the running code, per this project's ground-in-the-environment-first rule, because B4's correctness rests entirely on them.

1. The cache invariants (Component 3) — non-negotiable

  • U — every non-NULL serverUlid is UNIQUE. Duplication is structurally impossible at rest; many NULLs (un-acked rows) coexist.
  • A — W2 collapse + W3 upsert each run in one transaction(); watchers emit only post-commit, so the UI never sees an intermediate two-row state.
  • O — the outbox is a query (serverUlid IS NULL AND deliveryState != failed), not a table. Single-row discipline: reconcile in place, never mint a second row.
  • R — reply resolution joins on serverUlid, never clientTempId.

And the cache already makes the hard race atomic: when history inserts a server row R_u before the ack lands, reconcileAck's collapse path (drift_cache.dart:160-189) DELETEs R_u then UPDATEs the optimistic row R_c, in one transaction. B4 does not re-solve this — it relies on it.

Self-echo: the collapse path is the steady state, not the edge. Verified hub.py:44: fanout delivers to every connection subscribed to the channel, with no sender exclusion. The sender is subscribed (it is viewing the channel), so on a normal send it receives BOTH the ack(c→u) (ws.py:82) AND a MessageFrame(msg_id=u) of its own message (ws.py:89), in undefined order. Therefore the W2-collapse / W3-dedup race is not a reconnect curiosity — it runs on the happy path of every send. The design leans on this being atomic (it is, in the cache); the consequence for B4 is that the test plan must race ack-vs-self-echo in both orders for an ordinary send, not only across a reconnect. (A future gateway optimization could suppress self-fanout, mirroring the bus echo.mark_sent path; until it does, B4 must assume self-echo.)

2. The gateway re-send contract — the invariant B4's drain depends on

B4's reconnect re-sends un-acked outbox rows. "Un-acked" includes the message arrived; only the ack was lost. If a re-send minted a second server ULID, history would later pull both — the worst failure, sneaking in below invariant U (the cache can't see a duplicate the server created). So B4 rests on the gateway being idempotent. Confirmed by reading aiko_chat_gateway:

  • messages_service.py:45-46 — a resent (channel_id, client_msg_id) returns (existing, False): the same row, same ULID.
  • models.py:59 — backed by a DB UniqueConstraint("channel_id", "client_msg_id"): enforced at storage, not just app logic.
  • ws.py:82 — the ack always carries row.id, so a resend returns the same ULID and the original created_at.
  • ws.py:84 — bus-publish + fanout are gated on if created:, so a resend acks the sender but does not re-fanout: no duplicate delivery to other peers.
The elegant consequence. Because the gateway is idempotent and the cache's collapse path is the atomic backstop, drain-first is an optimization, not a correctness requirement. It shrinks the window in which history can beat an ack; it does not have to succeed for correctness. That is what licenses the best-effort-with-timeout await the AMR reality demands.

Invariants B4 itself must hold (stated before the mechanism)

  • B-single — B4 is the only writer to the cache. Nothing else calls W1–W5. (Keeps the writer census closed; a 7th writer means re-audit U/A/O/R.)
  • B-live — the transport stream subscriptions (messages/acks/errors) are established once and outlive every reconnect, so no ack or inbound frame is ever dropped because a socket bounced. (The transport already guarantees its streams outlive reconnects — finding 3 in Component 1; B4 must not re-subscribe per-connection and orphan the old listener.)
  • B-order — on reconnect, drain is dispatched before history is fetched. Best-effort: a drained row whose ack times out does not block history; the collapse path covers it.
  • B-optimistic — sendMessage commits the optimistic row (W1) before the wire send, so an app kill between the two loses nothing: the row is in the outbox and re-sends on next connect.
  • B-noteleport — a retry (W5) preserves createdAt + localSeq; B4 never re-stamps time on retry. A message keeps its place in the timeline.

Mechanism — the four paths

W1 path — optimistic send

Future<void> sendMessage(String channelId, String body, {String? replyToId}) async {
  final tempId = _uuid.v4();
  final optimistic = Message(
    clientTempId: tempId,
    id: null,                                  // serverUlid NULL → in the outbox
    channelId: channelId,
    sender: _me,                               // local render-as-me; server re-derives from JWT (I5)
    body: body,
    replyToId: replyToId,
    createdAt: _clampToBottom(channelId),      // appears at the bottom immediately
    deliveryState: DeliveryState.sending,
  );
  await _cache.insertOptimistic(optimistic);   // W1 — COMMITS (derives localSeq) before the wire send
  _transport.sendMessage(OutgoingMessage(
    clientTempId: tempId, channelId: channelId, body: body, replyToId: replyToId));
}
  • _me — B4 needs the current user identity to render the optimistic row as "me" (the wire send carries no sender; the gateway derives it from the JWT, I5). So ChatRepository depends on the resolved AppUser (from auth). This is a real dependency edge to wire via Riverpod, not a free-floating singleton.
  • _clampToBottom — the optimistic createdAt is max(now, newestCreatedAt + 1ms) for the channel, so a skewed client clock can't park a pending message in the past. localSeq (W1: MAX+1) is the within-same-millis tiebreak that keeps rapid sends in compose order.
  • Named tradeoff — ack may settle a message slightly upward under forward skew. If the client clock is ahead, the clamped optimistic createdAt > server time; W2 stamps the authoritative server time on ack, so the row can move up the timeline at the moment it turns sent. This is accepted, not a bug: server time MUST win (it is the only ordering all clients agree on — a client-time-wins rule would desync the AMR room). The movement is bounded by the clock skew and happens once, at ack. It is named here and pinned by a test (ack-reorder) so it can't masquerade as a regression. The stronger "no teleport" guarantee (B-noteleport) applies to retry, where there is no new server truth to honor.
  • No try/catch around the wire send — and it is intentionally not awaited. The transport contract is String sendMessage(OutgoingMessage) (chat_transport.dart:51): synchronous, returns the clientTempId, and never throws — deliverability is signalled out-of-band via the acks/errors/connectionState streams and the outbox, never via the call returning. So there is no Future to await and no exception to catch; coupling send-success to the call would be a category error.

W2 / W3 / W4 paths — the stream handlers (established once, B-live)

void _wire() {
  _acks.listen((a)  => _onAck(a));          // W2
  _msgs.listen((m)  => _cache.upsertInbound(m)); // W3 — fanout echo + others' + history-via-stream
  _errs.listen((e)  => _onError(e));         // W4
  _conn.listen((s)  => _onConnState(s));     // reconnect choreography
}

Future<void> _onAck(AckResult a) async {
  final outcome = await _cache.reconcileAck(a.clientMsgId, a.msgId, _parse(a.createdAt));
  _completeAckWaiter(a.clientMsgId);                 // unblock a drain waiter (see reconnect)
  if (outcome == AckOutcome.orphaned) {
    // Unreachable in Phase 1 BY CONSTRUCTION: W1 always persists the optimistic
    // row before the wire send, and there is no local delete (W6 is Phase 2). So
    // an orphaned ack means corruption or a bug, NOT a routine path. Policy:
    //   1. log loudly (telemetry) — this should never fire;
    //   2. do NOT attempt a targeted backfill — the ack carries no channelId
    //      (AckResult has only clientMsgId/msgId/createdAt), so a targeted
    //      getHistory is impossible from an ack alone. The acked message is on
    //      the server, so the NEXT reconnect's full history sync ingests it via
    //      W3 regardless. No message is lost; only a (near-impossible) corruption
    //      is surfaced rather than silently swallowed.
    _telemetry.orphanAck(a.clientMsgId, a.msgId);
  }
}

Why not just add channelId to the ack so targeted repair becomes possible? Because it would be machinery for an unreachable case. If orphaned ever does fire in production telemetry, the proportionate fix is then to add channel_id to the ack envelope (the gateway already holds it at ws.py:82, inside _handle_send which has frame["channel_id"]) and upgrade this branch to a targeted backfill. Until the telemetry says otherwise, the next-history-sync backstop is sufficient and simpler. Dijkstra: "Simplicity is prerequisite for reliability."

W3 / W4 — inbound and errors, with an explicit error taxonomy

W3 (upsertInbound) is a thin pass-through — the cache owns its atomicity (and, per self-echo above, this is where a message's own fanout lands). W4 (markFailed) needs a real error taxonomy, because "retry on next reconnect" infinite-loops on a terminal failure (an expired token retries forever, burning the robot's battery):

ErrorShapeB4 policy
Per-messagerefClientMsgId != null (e.g. no_channel, bad_frame)cache marks that row failed (only while serverUlid IS NULL); it leaves the outbox. UI offers W5 retry. No auto-retry.
Systemic — transientrefClientMsgId == null, code in {rate_limited, transient server}leave pending rows in sending; apply exponential backoff before the next drain (do not hot-loop); surface a single channel-level banner, not N per-message failures.
Systemic — terminalauth/token failure (or transport emits ConnectionState.unauthenticated)stop draining entirely. Route through unauthenticated (the transport's terminal state, chat_transport.dart:6) → the router redirects to login. No retry loop; pending rows stay sending and resume after re-auth.

The axis is transient vs terminal, not per-message vs systemic — a terminal systemic error (auth) and a transient systemic error (rate-limit) are not thermodynamically equivalent and must not share a policy. The backoff counter is per-channel and resets on the first successful drain.

The reconnect choreography — drain-first, timeout-bounded (B-order)

// --- Single-flight re-entrancy guard (BLOCKER fix, round 2) ---
Future<void>? _reconnecting;   // non-null while a choreography runs
bool _rerunRequested = false;  // a connected event arrived mid-flight

void _onConnState(ConnectionState s) {
  if (s != ConnectionState.connected) return;        // ignore disconnected/connecting/unauth
  if (_reconnecting != null) {                        // already choreographing →
    _rerunRequested = true;                           //   request exactly one re-run, don't spawn a second
    return;
  }
  _reconnecting = _runReconnect().whenComplete(() {
    final rerun = _rerunRequested;
    _rerunRequested = false;
    _reconnecting = null;
    if (rerun) _onConnState(ConnectionState.connected); // coalesced: at most one queued re-run
  });
}

Future<void> _runReconnect() async {
  _transport.subscribe(_subscribedChannelIds);        // additive, safe to repeat
  await _drainOutbox();                               // re-send un-acked, await acks BEST-EFFORT
  await _fetchDeltaHistory();                         // then reconcile against server truth
}

Future<void> _drainOutbox() async {
  final pending = await _cache.outbox();              // O: the query, in send order
  final waiting = pending.map((m) => m.clientTempId).toSet();
  _registerAckWaiters(waiting);                        // REGISTER FIRST (fixes the ack-before-registration race)
  for (final m in pending) {
    _transport.sendMessage(_toOutgoing(m));            // gateway is idempotent → safe re-send
  }
  // A row already acked between outbox() and now (or fast self-echo) is complete
  // immediately — don't burn the timeout waiting for an ack that already happened.
  await _resolveAlreadyAcked(waiting);
  await _awaitAcksOrTimeout(waiting, timeout: const Duration(seconds: 3));
}

Re-entrancy (BLOCKER, round 2). A flapping link can emit connected twice. The guard is single-flight with one coalesced re-run: while _reconnecting != null, a second connected sets _rerunRequested and returns; the in-flight choreography re-runs exactly once on completion. So at most one choreography runs at a time, and a bounce that arrives mid-history-paging is not lost (it queues the single re-run) but also cannot stack N concurrent paging loops. _onConnState is now synchronous up to the guard, so the check-and-set can't itself interleave.

Ack-wait race (MAJOR, round 2). Waiters are registered before sends are dispatched, and _resolveAlreadyAcked completes any whose row is already serverUlid != null in the cache — so an ack that lands in the dispatch window (fast local gateway, replayed stream, or self-echo) can't be missed and leave the drain burning the full timeout. _awaitAcksOrTimeout is a single Future.wait(...).timeout(...); on timeout we proceed to history — the collapse path guarantees correctness if an ack lands afterward. This is the one place the design deliberately accepts "good enough" because the backstop makes it safe, and it is now the only such place — the other four "good enough"s from round 1 are now specified.

The two upstream gaps this design surfaces

Grounding the design against the real interfaces revealed two seams that don't quite meet. Both are small, both are flagged here rather than papered over.

Gap 1 — reconcileAck returns void; B4 can't observe the orphan case

The cache's reconcileAck silently returns on rc == null (drift_cache.dart:144, "orphan ack — B4 schedules a persisted repair") — but with a void return, B4 cannot tell reconciled from collapsed from orphaned. Proposed change: return an enum.

enum AckOutcome { reconciled, collapsed, orphaned }
Future<AckOutcome> reconcileAck(String clientTempId, String serverUlid, DateTime serverCreatedAt)

This adds no writer (census intact), is trivially testable, and turns a silent defensive branch into an observable contract. In Phase 1 orphaned is unreachable by construction — W1 always persists before the wire send and there is no local delete (W6 is Phase 2). So orphaned is a loud "this should never happen" telemetry signal, not a routine path. B4 does not attempt a targeted backfill (the ack has no channelId — see the round-1 BLOCKER); the acked message is on the server, so the next reconnect's full history sync ingests it via W3 regardless. The enum makes the impossible case observable; it does not pretend to repair it.

Gap 2 — reconnect wants a forward delta; getHistory only paginates backward

The plan says reconnect does getHistory(after=lastKnownServerId). The shipped ChatRestApi.getHistory(channelId, {before, limit}) only paginates backward from newest (a before ULID cursor; messages_service.py:107-120 confirms Message.id < before). There is no after.

Phase-1 resolution with a termination proof (MAJOR, round 2). The cursor basis is the server ULID total order — verified at messages_service.py:112-117: history is WHERE Message.id < before ORDER BY Message.id DESC, i.e. the gateway orders strictly by ULID, and ULIDs are monotonic + globally unique. So "what did I miss?" is exactly "all ULIDs > lastKnownServerId", and lastKnownServerId = the MAX serverUlid currently in the cache for the channel.

String? before = null;                          // start at newest
while (true) {
  final page = await _rest.getHistory(channelId, before: before, limit: 50);
  for (final m in page.messages) _cache.upsertInbound(m);   // W3 dedups on serverUlid
  final oldest = page.messages.isEmpty ? null : page.messages.first.id;  // ascending → first = oldest
  if (page.nextBefore == null) break;            // (a) reached the start of history
  if (oldest != null && oldest.compareTo(lastKnownServerId) <= 0) break;  // (b) overlap: gap closed
  before = page.nextBefore;                       // else page further back
}

Termination: each iteration strictly decreases before along a finite total order, so the loop halts at (a) the oldest message or (b) the first page whose oldest ULID is ≤ lastKnownServerId — the point where the fetched window provably overlaps what the cache already holds, so no ULID in (lastKnownServerId, newest] can remain unfetched. This is correct for >50 offline messages (it keeps paging) and cheap for the common case (one page, immediate overlap). It does not rely on created_at ordering at all — only the ULID total order — sidestepping the skew/divergence Carnot flagged. Edge: if the cache is empty (lastKnownServerId null — fresh install / first open), condition (b) never fires and the loop pages to the start, which is the correct "load all history" behavior, bounded by (a).

Flagged for later: a true after cursor (forward delta) on the gateway + ChatRestApi is the scalable form; file it when channel volume or multi-channel fan-in makes the "fetch newest page" approach wasteful. Not Phase 1.

Edge cases the reviewer should attack

  • Lost-ack re-send — message arrives at gateway, ack lost. Reconnect drains → re-send → gateway returns SAME ULID (idempotent) → W2 stamps it. Then history fetch returns the same message → W3 dedups. One row. Attack: does B4 ever treat the re-send's ack as a new message?
  • History beats the re-send's ack — drain dispatched, but the 3s timeout fires before the ack; history fetch runs and inserts R_u; the ack lands afterward → cache collapse path → one row. Attack: is the collapse path actually reached, or does B4 swallow the late ack?
  • App killed between W1 commit and wire send — optimistic row persisted, never sent. Next connect: it's in the outbox, drains, sends, acks. No loss. Attack: does B4 re-clamp createdAt or bump localSeq on this re-send (it must not — B-noteleport)?
  • Double connected events — flaky link emits connected twice. Resolved (round 2): the single-flight guard (_reconnecting + coalesced _rerunRequested) runs at most one choreography and queues at most one re-run. Attack: can a third event mid-run, or a bounce during history paging, still stack concurrent paging loops or lose the final reconcile?
  • Send while disconnected — W1 commits, wire send is a no-op (socket down). The row sits in the outbox until reconnect drains it. Attack: does the UI correctly show sending (not failed) while offline?
  • Systemic error during drain — Resolved (round 2): the error taxonomy splits transient (rate-limit → backoff, stay sending) from terminal (auth → stop draining, route through unauthenticated). Attack: does a terminal error mid-drain leave any waiter hanging on the 3s timeout, or any row stuck where backoff never resets?
  • Orphan ack (Gap 1) — force rc == null (e.g. an ack for an unknown clientMsgId). Attack: does B4 lose the message, or backfill it?

Test plan (ATDD — written and RUN before impl)

The hard-won lesson from Component 3: a single flutter test caught the W2 statement-order bug that 4 cage-match rounds across 3 model families reasoned right past. So every claim about ordering, timing, or atomicity below is a test, made to fail for the right reason first — not a review argument. Substrate: a real DriftCache on NativeDatabase.memory() + a new FakeChatTransport implementing the ChatTransport interface with test-driven emitAck/emitMessage/emitError/emitConn + a recorded sendMessage log, and a mocktail ChatRestApi. (The existing fakes are at the WebSocketChannel level — too low for B4; we fake the interface, per plan §B6.)

  • W1 sendMessage → cache has one sending row, serverUlid==null, in the outbox; transport recorded one send with the matching clientTempId.
  • W1 order commit-before-send, with the real cache (MINOR, round 2): assert ordering via observable state, not a cache spy — when FakeChatTransport.sendMessage is first called, query the real cache and assert the optimistic row is already present (committed). Keeps the integration property; no mock cache.
  • W2 happy (ack only) send → emitAck(sameTempId→u) → one row, serverUlid==u, state==sent, createdAt == server time.
  • self-echo A real happy path (round 2): send → emitAck(c→u) THEN emitMessage(self-echo u) → exactly one row (echo dedups via W3 on serverUlid).
  • self-echo B reversed order: send → emitMessage(self-echo u) THEN emitAck(c→u) → exactly one row (cache collapse path). Both orders asserted — self-echo races the ack on every send.
  • ack-reorder (round 2) forward-skewed clock → optimistic createdAt > server time → on emitAck the row's createdAt becomes server time (named tradeoff: settles upward, deterministically; pins the behavior so it can't look like a regression).
  • collapse send → emitMessage(history row u) (W3) → emitAck(c→u) → ONE row, clientTempId==c, serverUlid==u, server fields. (Cache-owned, but B4-driven ordering — pin it end to end.)
  • disconnect→reconnect→noDup the crux test. send (row pending) → emitConn(disconnected) → emitConn(connected) → assert transport re-sent the pending row → emitAck for it → fetch history returns the same message → exactly one row, state==sent.
  • lost-ack as above but the FIRST send's ack never arrives; only the re-send's does (same ULID) → one row, no dup.
  • drain-first order reconnect with one pending row + history containing it → assert transport.sendMessage (drain) is invoked before rest.getHistory (B-order).
  • ack-timeout reconnect, drained row's ack never arrives → after the timeout, getHistory still runs (reconnect does not hang); history inserts the row; late ack → collapse → one row.
  • no-teleport failed row → reconnect drain re-sends it → its createdAt + localSeq unchanged (B-noteleport).
  • re-entrancy (round 2) two emitConn(connected) in quick succession while the first choreography is still paging → exactly one concurrent choreography (assert getHistory is not called concurrently); the second event coalesces into exactly one queued re-run, not N. A third event during the run still yields one re-run.
  • ack-already-acked (round 2) reconnect drain where the row's ack arrives in the dispatch window (or via self-echo) → the waiter completes immediately, drain does NOT burn the full timeout (assert wall-clock < timeout).
  • offline-send emitConn(disconnected) → sendMessage → row is sending (never failed) and queued; later connected drains it.
  • B-live multi (round 2) drive THREE disconnect/reconnect cycles, then emitAck once → assert the reconcile fires exactly once (a doubled/orphaned listener would fire it N times, masked by idempotent cache writes — so count effects, not just final state). Include a repository disposal → assert no handler fires after dispose.
  • systemic-terminal (round 2) emitConn(unauthenticated) (or an auth error frame) during drain → draining STOPS, no retry loop; pending rows stay sending; the unauthenticated state is surfaced (router redirect seam).
  • systemic-transient (round 2) rate_limited systemic error → pending rows stay sending, next drain is backed off (not hot-looped), one channel banner.
  • orphan emitAck for an unknown clientMsgId → AckOutcome.orphaned observed → telemetry fires, NO crash, message not lost (next history sync would ingest it). Asserts the impossible case degrades safely.

"Done" for B4

A message survives the full lifecycle — send → disconnect → reconnect → ack — with no duplicate, no loss, and a correct DeliveryState + timeline position at every step (retry preserves createdAt+localSeq, no teleport). Verified by tests that induce disconnection mid-flight and confirm the outbox drains and reconciles to exactly one row. Process-done (tests green, merged) is not capability-done: the last gate is the lifecycle exercised against the real gateway on a real flaky link — the AMR reality — not just the fakes.
