Design 10 — The island/gateway semantic split

The codebase uses one word, gateway, for two different things: the sovereign node (who/where — the unit of federation and choice) and its protocol edge (how — the substrate-agnostic /v1/* seam). This is not a rename to apply; it is a distinction the code failed to make, and the code's own comments already half-know it. This doc proves the split from a census, classifies every reference, and sharpens the open decision from a yes/no into the two precise questions the island tab actually has to answer.

Status: app-side preparation — NOT a rename yet. The actual rename is blocked, deliberately, on two gates (see §6): the island session's canonical decision (claude-tasks#1760, project:aiko-chat-island, still OPEN) and the team group's naming reply. The canonical concept + wire contract live island-side (peer-owned); an app-only rename creates app-says-island / island-says-gateway drift. This doc is the work that turns the eventual rename into a clean coordinated push instead of a cold start.
Companion to Design 06 (federation north-star) and Design 05 (community/discovery). The vocabulary carved here is the one every future federation feature will reason in.

1 · The thesis — an island has a gateway

An island is the sovereign node: identity, community, users, operator, data — the unit of federation and of the user's choice. A gateway is that node's protocol edge: the substrate-agnostic /v1/* REST + WS seam the app speaks. An island has a gateway the way a landmass has a front door. The app's lib/ contains zero matrix/mosquitto references — it only ever speaks the edge; the island translates that edge to whatever substrate it runs. So the split is real, load-bearing, and asymmetric: the app knows gateways intimately and islands only by name.

This matters because Design 06's whole federation model is identity that is owned by no gateway. You cannot express "owned by no gateway" cleanly while the word gateway also means "the node that owns things." The vocabulary has to carve the node away from its edge before the federation vision has words to be built in. This is the smallest possible unblocking move for the north-star: not code, names.

2 · The evidence — the code already knows

The strongest sign a distinction is real is when the code narrates it in comments while the symbols lag. Two smoking guns, both in lib/features/settings/domain/server_entry.dart:

Gun 1 — the presets are described as islands, five times over

// kGatewayPresets — the built-in seeds...
// "MULTIPLE real internet islands are bundled (not just one), so bootstrap
//  discovery survives any single one being down..."
// "Every island serves the FULL peer directory from /v1/gateways, so reaching
//  ANY one of these teaches the app about all the others..."
// example description: "Imagineering's flagship island"

The constant is named kGatewayPresets and the type is ServerEntry, but the concept the comments reach for — every single time — is island. The code knows what these are. Only the names disagree.

Gun 2 — the entire split fits in one sentence

"Every island serves the FULL peer directory from /v1/gateways."

The island (node) serves; the /v1/gateways (edge) is the how. Node-concept and edge-concept co-occur in a single clause, each already named correctly for its role. The distinction isn't something we're imposing — it's already there in the prose, waiting for the symbols to catch up.

The counter-check — GatewayConfig is pure edge

lib/app/config.dart's GatewayConfig is 100% protocol edge: "where the gateway lives", an HTTP base, a WSS URL derived from it (http→ws) so the two can't drift, $base/v1/... composition. There is nothing sovereign-node about it. It should stay gateway — and the fact that it sorts so cleanly to the edge pile, while kGatewayPresets sorts so cleanly to the node pile, is the proof the two concepts are genuinely separable.

Rule of thumb that resolves every case: if it's about which node → island; if it's about how to talk to a node → gateway. A URL and a REST client are "how"; a directory, a picker, a preset, a seed store are "which".

5 · Three tranches by gate

The key finding of this census: the blockage is not monolithic. It layers into three tranches, each gated differently. Treating them as one all-or-nothing rename is what made this feel more blocked than it is.

A · Protocol-edge — stays gateway, no change, ever

GatewayConfig, GatewayRestApi, gateway_transport, kDefaultGatewayBaseUrl, the auth clients. These are correctly named today. The refactor's first act is to affirm they don't move — otherwise a naïve find-replace would wrongly island-ify the edge and destroy the very distinction we're drawing.

B · Node-identity Dart symbols — mechanically app-autonomous, semantically gated

ServerEntry→IslandEntry, kGatewayPresets→kIslandPresets, the directory / seed / picker files. These are internal Dart symbols — renaming them touches no wire byte (the persisted JSON already uses neutral snake_case keys like base_url, display_name). Mechanically the app could do this alone. But it shouldn't yet, for a subtle reason: the rename picks a noun, and the noun is the island tab's call (§6). Rename to IslandEntry before #1760 settles, and if the canon lands on "Node" or "Server" we churn twice. The distinction is app-verifiable; the word is not.

C · The wire contract — island-coordinated + compat window, blocked hard

/v1/gateways (path) and the 'gateways' response key are the real peer-owned artifact. This is an edge-named path returning node-identity data — the perfect crystallization of the whole conflation. The app only consumes it; it cannot rename it. And renaming it is not free even for the island: it needs a compat window (serve both /v1/gateways and, say, /v1/islands; accept both JSON keys) so old app builds don't break.

Tranche C splits into a reversible half and an irreversible half — and only the reversible half is shipped here. The two are not the same commitment:

  • Response-key widening (reversible, done now). The client's accepted-key list now leads with 'islands' ahead of the legacy 'gateways' (['islands','gateways','servers','entries', 'directory']). Note this is a canonical-precedence prepend, not a passive append to the fallbacks: when a compat-window peer double-serves both keys, 'islands' wins — a behavioural commitment, guard-contract tested, and fail-soft (an empty or all-malformed 'islands' does not shadow a populated 'gateways'). Reversible: drop the key from the list and nothing breaks.
  • Request-path migration (irreversible, NOT done, still island-gated). The app still requests $base/v1/gateways (gateway_directory_provider.dart). If the island renames the path, the app must change what it sends — and a sent path can't be widened the way a parsed key can (you request one URL). That later app-side edit is real work this PR does not pre-pay; it waits on the island publishing the new path + its own dual-serve window.

So the down-payment is genuinely safe to ship now because the reversible half only widens what we accept; the irreversible half (what we request) is untouched and stays island-gated.

Do not rename the /v1/gateways path app-side under any circumstance — it's a request the app sends. Changing it without the island serving the new path is an instant break. Tranche C moves only when the island moves.

6 · The two gates — and why even the internal rename waits

The rename is blocked on two things. Both must be open before Tranche B ships:

  1. The island session's canonical decision — claude-tasks#1760 ("own the island-vs-gateway distinction + wire contract", project:aiko-chat-island). Status as of this writing: OPEN. This is where the canonical noun and the /v1/gateways migration are decided. The concept lives island-side by design (Design 06 §identity: identity owned by no gateway → the node concept is the island's to name).
  2. The team group's naming-alignment reply — Nick's message to the group; no reply yet at last check.

Even Tranche B (pure internal symbols) waits, because the noun is not the app's to coin. This is the non-obvious part: mechanical autonomy ≠ semantic autonomy. The app can rename the symbols without touching the wire, but it can't choose what to rename them to without re-centralizing a decision Design 06 deliberately pushed to the node.

7 · The sharpened handoff — two decisions, not a yes/no

The census converts the fuzzy question — "are these gateways or islands?" — into two precise decisions the island tab can actually make. That's the whole value of doing this app-side prep now:

Decision 1 — the canonical noun for the sovereign node. The code's comments already vote island (5× in one file). Is the canon Island, or something else (Node, Server, Home)? This unblocks Tranche B verbatim. Recommendation: Island — it's what the code already reaches for, and it carries the sovereignty connotation Server/Node lack.

Decision 2 — the /v1/gateways wire migration. Three options: (a) rename to /v1/islands with a compat window (serve both, accept both JSON keys) — cleanest, matches the concept; (b) keep /v1/gateways as a deliberate historical name for the edge's directory endpoint and document that the edge serves the island directory — zero migration, permanent minor conflation; (c) defer. This is purely the island's call; the app will follow whatever compat contract it publishes.

When both gates open and both decisions land, the app-side work is a single clean coordinated PR: affirm Tranche A, apply Tranche B with the chosen noun, and — if Decision 2 picks path-rename (option a) — point the request path at /v1/islands (Tranche C's irreversible half; the reversible key-widening half is already shipped). No cold start, no guesswork — this doc is the plan, pre-verified against the real code.
