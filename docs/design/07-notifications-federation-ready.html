Design 07 — Notifications (federation-ready)

Push notifications are the one place the federation thesis (Design 06: better by subtraction) hits a wall subtraction can't remove: the mobile platforms weld push to a single app identity. You cannot decentralise your way around APNs. The answer — proven at scale by Matrix/Element — is to accept exactly one deliberately-centralised component (a push relay) and keep everything else federated. This doc is the model, the Element teardown it's grounded in, and the one cheap principle to adopt now so launch-simple notifications never need a federation rewrite.

Status: design. The launch slice (§2, single gateway) is buildable now; the federation slice is federation-ready structure, not a v1 build. Companion to 05 (communities) and 06 (federation). Captured 2026-06-30.
Grounded fact: the app has zero push infrastructure today (no firebase_messaging, no APNs entitlement, no aps-environment) — this is greenfield, so the cheap principle costs ~nothing now.
Legend: APP client · GATEWAY home gateway · RELAY the push gateway (new component). Confidence: solid grounded · plausible coherent-unvalidated · thin genuinely open.

1. The thesis — the one wall subtraction can't remove

Everywhere else, federation gets simpler by moving cleverness to identity (06). Notifications are the exception: APNs and FCM bind push delivery to one app identity — one bundle id, one signing key, owned by the app vendor. No amount of sovereign-key cleverness changes that a third-party gateway physically cannot push to your app. So the design move here is not "decentralise it" — it's quarantine the centralisation: one small relay component holds the platform credentials, everything else stays federated, and you copy the pattern the most mature federated chat on earth already proved.

2. What's true today — greenfield + one gateway

solid Verified 2026-06-30: there is no notification code in the app (no messaging dep, no APNs entitlement). And there is exactly one gateway (chat.imagineering.cc), to which the app is compile-time bound. So two things follow:

  • Launch notifications are trivially easy. One gateway → wire APNs/FCM straight from it. None of the federation complexity below bites until there is a second gateway.
  • It's the cheapest possible moment to adopt the federation-ready shape (§10) — there is no migration, because there is nothing built. This is the 06 §8 lesson (namespace globally from day one) applied to push.

3. The collision — push is welded to app identity

solid To deliver a push to the Aiko app on a device you need both:

  1. the device's push token (APNs device token / FCM registration token), and
  2. the app's signing credentials — one APNs auth key + one FCM project, both welded to the single bundle id cc.imagineering.aikoChatApp / cc.imagineering.aiko_chat_app, both owned by the vendor's store accounts.
A third-party gateway operator has neither. They don't hold your users' device tokens (those live at each user's home gateway), and they certainly don't hold your app's APNs key. So a federated peer gateway cannot push to the Aiko app. And you cannot fix this by handing every operator your APNs key — that key can notify (and therefore spoof/phish/spam) every Aiko user on earth. Push credentials are an unshareable, vendor-bound secret. This is the wall.

4. The relay topology already contains it

solid The 06 model's client rule is the thing that keeps this tractable: "the app only ever talks to its own home gateway over one WSS connection; a channel homed elsewhere is reached through the home gateway's relay" (06 §3). Therefore:

  • The user's home gateway is the single notification origin. A message in a channel homed on a remote gateway is relayed to the user's home gateway (06 §6 relay-and-verify); the home gateway then decides whether to notify and triggers the push. Only home gateways notify — never every channel-home gateway.
  • So push fan-out follows the same single-home discipline B4 already depends on. There is exactly one party responsible for notifying you: your home gateway. No coordination, no duplicate pushes from N gateways.

This contains the problem to: "home gateways — some of which are third-party — need to cause a push to the Aiko app." Which §3 just said they can't do directly. Hence the relay.

5. The push relay (the Sygnal pattern) RELAY

solid Split the job in two. The home gateway decides who/what to notify (it knows the user's notify preferences and holds their device registry). A separate push relay holds the platform credentials and does the actual send. The vendor (Aiko) runs exactly one relay; every gateway — yours or a third party's — calls it.

  message on a channel (home gateway H, or relayed in from a peer)
        │
        ▼
  HOME GATEWAY        evaluate notify rules for the user; look up their device pushkeys
        │  POST /push/notify  { pushkey, app_id, event_ref, counts, [content?] }
        ▼
  PUSH RELAY          holds the APNs key + FCM project; builds + signs the platform push
        │  APNs / FCM
        ▼
  DEVICE              wakes; (optionally) fetches the event from its home gateway; renders
What is and isn't centralised. The relay re-centralises only the platform-credential-holding and the send. It does not hold accounts, channels, history, or identity — those stay federated across gateways. Notify logic (rules, mute, mentions) stays on each home gateway. The relay is a dumb, stateless-ish fan-out box that turns a notify request into an APNs/FCM call. That is the smallest centralised surface that the platform wall forces — and no smaller is possible for the official app on iOS (§7).
Wire shapes (modelled on Matrix's push API — copy, don't invent) 
// 1) The app registers a "pusher" with its HOME gateway on startup / token refresh.
//    (Matrix: POST /_matrix/client/v3/pushers/set)
POST /v1/pushers/set
{
  "pushkey":     "<APNs device token | FCM token | UnifiedPush endpoint>",
  "app_id":      "cc.imagineering.aikoChatApp.ios",   // platform-specific
  "kind":        "http",
  "push_relay":  "https://push.imagineering.cc",       // the vendor relay URL
  "device_id":   "<this device>",
  "append":      true,                                  // keep other devices' pushers
  "data":        { "format": "event_id_only" }          // privacy mode, see §9
}

// 2) The home gateway, on a notify-worthy event, calls the relay.
//    (Matrix: POST /_matrix/push/v1/notify)
POST https://push.imagineering.cc/v1/notify
{
  "notification": {
    "event_id":   "<ulid>",
    "channel_id": "<global id: id@homeGateway>",
    "counts":     { "unread": 3 },
    "devices":    [ { "app_id": "...", "pushkey": "...", "data": {"format":"event_id_only"} } ]
    // content (sender, body) included ONLY if the pusher did not request event_id_only
  }
}

6. How Element does it (the reference teardown)

plausible on exact endpoint versions; solid on the architecture. Element is the flagship Matrix client and solved this exact problem — its design is the prior art. Three components: the homeserver (Synapse) decides who/what; Sygnal (the push gateway) holds the app credentials and sends; Element registers a pusher and renders.

  1. Pusher registration. Element gets a platform token (APNs / FCM / UnifiedPush), then POST /_matrix/client/v3/pushers/set to its homeserver with the pushkey + the push-gateway URL to use.
  2. Push rules. On a new event the homeserver evaluates the user's push rules (mentions, keywords, per-room mute) and decides whether to notify. Notification logic is per-homeserver — decentralised.
  3. Homeserver → Sygnal. If notifying, the homeserver POSTs /_matrix/push/v1/notify to the push gateway with the pushkey + event reference (+ optionally content).
  4. Sygnal → APNs/FCM. Sygnal holds Element's APNs key + FCM project and sends a usually-silent push.
  5. Device fetch + decrypt. iOS wakes a Notification Service Extension; Android a background service. It calls back to the homeserver to fetch the event, decrypts it locally (E2EE), and posts the real notification. Content never had to pass through Sygnal.
The punchline that answers the whole question: Sygnal holds Element's credentials, not the homeserver's. Run a fully sovereign, self-hosted Synapse and want it to notify the official Element app? Your homeserver POSTs to matrix.org's / Element's Sygnal — the only push gateway on earth holding Element's APNs key. Your server is sovereign; your push still routes through the app vendor's central relay. Sovereignty stops at the push gateway. The most mature federated chat in existence makes exactly this compromise — accepting one central relay rather than fighting an unwinnable fight with APNs. The only full escape is shipping your own app (own bundle id, own Sygnal), which is precisely what Aiko is.

7. The iOS wall & the Android escape APP

PlatformDecentralised push possible?Mechanism
AndroidYesUnifiedPush: the user picks their own push distributor (self-hosted ntfy, etc.), bypassing both FCM and the vendor relay. Element's F-Droid build supports it. Aligns with the sovereignty ethos.
iOSNoApple welds push to APNs + the app entitlement. No UnifiedPush equivalent exists or can exist. iPhone users must route through the central relay, full stop.

So the honest shape: the sovereignty story has an iOS-shaped asterisk. Android can be fully decentralised (UnifiedPush) for users who want it; iOS cannot. This is not an Aiko failing — it is a platform constraint that binds Element and every other federated client identically. Name it as a named tradeoff, support UnifiedPush on Android as the purity option, and run the central relay for everyone else.

8. The sovereign-identity twist — device registry GATEWAY

plausible Push is device-bound (a token identifies a device); 06 identity is key-bound (portable, multi-device). So the home gateway must keep a per-identity device registry: which devices does this sovereign identity have, and what is each one's current pushkey? On a notify, it fans out to all of the identity's registered devices (each pusher, like Matrix). Implications:

  • Device registration is keyed by (sovereign identity, device), not by gateway-local user row — the same day-one discipline as global IDs (06 §8). A device's pusher moves with the identity if the user re-homes.
  • Token rotation (APNs/FCM tokens expire) → the app re-registers its pusher; the relay reports failed/expired pushkeys back so the home gateway prunes them (Matrix does this — "rejected" pushkeys in the notify response).

9. Privacy — notify without leaking content

solid A central relay is a metadata chokepoint, so minimise what flows through it. Adopt Matrix's event_id_only mode as the default:

  • The notify to the relay carries only event reference + counts + pushkey — no sender, no body.
  • The push wakes the app; the app fetches the event from its own home gateway and renders the real notification (iOS NSE / Android service). Message content never transits the relay.
  • This also future-proofs E2EE: if bodies are ever end-to-end encrypted, the relay (and the home gateway) can't read them anyway — the device decrypts.

Residual the relay still sees: that a given pushkey is being notified, and when (timing/volume metadata). Unavoidable for any push relay; documented, not hidden.

10. Phasing & the one cheap principle

PhaseWhat shipsFederation cost later
N1 — launch (one gateway)APP APNs/FCM token + register pusher with the gateway; NSE/background handler that fetches+renders. GATEWAY notify-rule eval + a notify call. The "relay" can be an in-process function on the one gateway — but behind a relay-shaped interface.—
N2 — extract the relayLift the APNs/FCM send into a RELAY standalone service (push.imagineering.cc); the gateway calls it over HTTP instead of in-process. No app change.Low — same interface.
N3 — federation-readyThird-party gateways point their notify at the same vendor relay; device registry keyed by sovereign identity; UnifiedPush option on Android.Low — the seam already exists.
The one cheap principle (06 §8, one layer up): at launch, shape notifications as app → home gateway (pushkey + relay URL) → relay → device, with the APNs/FCM send behind a separable relay interface even if that "relay" is just an in-process call on the single gateway. Key the device registry to sovereign identity + device, not a gateway-local user row. That costs ~nothing now and means federation never forces a notification rewrite — you just point peer gateways at the same relay. Same discipline as global IDs; the expensive thing is retrofitting a flat, gateway-bound push table under a live user base.

11. The hard problems (honest map)

  • thin The central relay is a single point of control/failure. A "sovereign" network with one mandatory vendor relay is a real philosophical compromise — and an availability one (relay down → no iOS pushes network-wide). Mitigate with redundancy + UnifiedPush-on-Android, but iOS has no escape. Accept and name it (Element does).
  • plausible Trust between peer gateway and relay. A third-party home gateway calls the vendor relay with "notify pushkey X." A malicious/compromised gateway could spam-notify its own users, or probe pushkeys. The relay must authenticate calling gateways and rate-limit per gateway — and a gateway can only ever reach the pushkeys it was given (its own users'), never enumerate others'. Blast-radius design, pre-federation.
  • plausible Notify-rule placement across a relayed channel. When a message is relayed in from a peer gateway, the home gateway evaluates notify rules — but does it have enough context (mentions of the user, the user's mute state for that remote channel)? The relay-and-verify payload (06 §6) must carry enough for the home gateway to run rules locally. Solvable; needs the federation message shape to include it.
  • plausible Metadata at the chokepoint (§9). Content is kept off the relay, but timing/volume per pushkey is visible to it. Inherent; documented.

12. Prior art & references

  • Matrix push gateway API — the /_matrix/push/v1/notify contract and the pushers/set client API. The shapes in §5 are modelled directly on it. (spec.matrix.org/latest/push-gateway-api)
  • Sygnal — Matrix.org's reference push gateway; the concrete "vendor runs one relay holding the app's APNs/FCM creds" implementation. (github.com/matrix-org/sygnal)
  • UnifiedPush — the decentralised push standard (Android); the §7 escape hatch. (unifiedpush.org)
  • APNs token-based provider auth + FCM HTTP v1 — the platform send the relay performs.
  • Push rules — Matrix's per-user notify-decision model evaluated server-side; the reference for §5's "home gateway decides who/what."

Companion: this design's relay is a new RELAY service (vendor-run, peer to the gateways) — a project:aiko-chat-island-adjacent build, handed off like 05/06's gateway halves. The app half (N1: token + pusher registration + NSE/background render) is the only launch-path piece, and it is small. Everything beyond N1 is federation-ready structure, not a launch commitment.
