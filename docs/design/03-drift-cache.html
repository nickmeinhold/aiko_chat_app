Design 03 — Drift cache

The local store B4 writes through — schema, the three writer paths, and the one uniqueness invariant that makes the app's worst failure impossible.

Component scope (Phase 1, text-only): the on-device SQLite store (drift package) that holds the one coherent local truth the UI renders and the repository/reconcile engine (B4, Component 4) mutates. Tables + DAO + reactive per-channel streams. No reconcile orchestration here — that is B4. This component owns storage, atomic operations, and invariant enforcement; B4 owns policy (when to drain, how to order, which DeliveryState transition to apply). Drawing that line is the whole point: if the cache enforces "one row per logical message" at the write boundary, B4 cannot duplicate even with a buggy ordering.

Naming: the package is literally drift (formerly moor) — reactive SQLite with build_runner codegen and Stream queries (.watch()). "Drift cache" = the local store; the homophone is a coincidence we lean into.

Reviewed-as-one because the schema + the three writer paths + the uniqueness invariant are a single bug surface; the table columns alone are trivial.

Source of truth — the gateway wire, re-verified by reading the running code

The schema is shaped by what ws.py actually does on the wire — read, not assumed (aiko_chat_gateway/.../realtime/ws.py:66-89):

  1. Ack precedes fanout, on one connection. _handle_send sends the ack (line 82) then hub.fanouts the message_frame (line 89). The sender, if subscribed, receives both: an ack {client_msg_id → msg_id} first, then a message frame carrying only msg_id (no client_msg_id — message_view omits it).
  2. The server is idempotent on client_msg_id. create_outbound returns (row, created). The ack fires unconditionally (line 82); bus-publish + fanout run only if created (line 84). So an outbox re-send is safe: same msg_id back in the ack, no second echo.
  3. The ack is the only frame carrying both ids. It is the sole join point between the client temp id and the server ULID. Everything reconcile depends on this one frame.

These three facts (verified against the running gateway, per Kelvin's ground-in-environment-first shift) are what the schema and writer paths below are built to satisfy.

The duplicate-row trap (why this component exists)

Message.clientTempId is the durable PK, and for server-origin rows clientTempId == serverUlid (message.dart:141). So the same logical message wears two different PKs across its life:

phaseclientTempIdserverUlid (id)deliveryState
optimistic (just composed)<uuid v4>nullsending
after our own ack<uuid v4><ULID>sent
its own fanout echo, naïvely inserted<ULID><ULID>sent

A naïve insert of the echo (or a history row for a message we sent) creates a second row keyed by the ULID — duplication. The defense is structural, not procedural:

Invariant U (the load-bearing one): serverUlid is UNIQUE. At most one row may ever hold a given ULID. Any write that would introduce a second is either deduped (drop) or collapsed (merge) — never inserted. The UNIQUE index is the write-level backstop that turns a raced dedup-check into a caught error instead of a silent duplicate. (Load/write must agree: the same predicate — "this ULID is not already present" — gates both the check and the constraint.)

Schema

Single messages table (the outbox is a query, not a table — see Invariant O), plus channels.

messages
  clientTempId   TEXT  PRIMARY KEY              -- durable, client uuid OR = serverUlid for inbound
  serverUlid     TEXT  NULL  UNIQUE             -- dedup authority; null until acked. ULID = time-sortable
  channelId      TEXT  NOT NULL  (indexed)
  senderUserId   TEXT  NULL
  senderKind     TEXT  NOT NULL                 -- SenderKind.wire; lenient decode on read
  senderLabel    TEXT  NULL                     -- nullable by wire contract (finding #4)
  kind           TEXT  NOT NULL                 -- MessageKind.wire
  body           TEXT  NOT NULL
  replyToId      TEXT  NULL
  createdAt      INTEGER NOT NULL               -- UTC unix millis (drift maps DateTime); provisional for optimistic
  localSeq       INTEGER NOT NULL DEFAULT 0     -- DB-DERIVED monotonic counter: SELECT COALESCE(MAX(localSeq),0)+1 inside W1's txn. 0 for inbound. NOT an in-memory counter (would reset on restart). Send-order tiebreak.
  deliveryState  TEXT  NOT NULL                 -- DeliveryState.wire

channels
  id          TEXT PRIMARY KEY
  name        TEXT NOT NULL
  kind        TEXT NOT NULL                     -- ChannelKind.wire; lenient decode
  aikoChannel TEXT NULL                          -- the mapped aiko channel (gateway sends it)

Ordering key: ORDER BY createdAt ASC, localSeq ASC, COALESCE(serverUlid, clientTempId) ASC over an index on (channelId, createdAt, localSeq, serverUlid). Three layers, each catching what the prior leaves ambiguous:

  • createdAt — the primary order; server time once acked, clamped client time while pending.
  • localSeq — the send-order tiebreak (round-3 fix). When several of my own optimistic rows clamp to the same createdAt (below), a bare clock tie would fall through to random clientTempId (uuid) order — so the user's own two consecutive sends could render reversed. The monotonic per-device localSeq preserves compose order among them. Inbound rows carry 0, so this layer is inert for them (their createdAt already separates them).
  • COALESCE(serverUlid, clientTempId) — final total-order tiebreak; also fixes the engine-dependent NULL sort of a bare serverUlid ASC (every row gets a non-null sort value). A ULID sorts lexicographically by time, so among acked rows this is authoritative.
Provisional clock is clamped, not trusted. An un-acked row's createdAt is the client clock, which can be grossly wrong (cage-match: "a device set to 2049" teleports the message pages away, then snaps back on ack). Clamp the optimistic createdAt to min(clientNow, lastKnownServerTime + δ) so a skewed clock can't fling the row out of the live view; the ack replaces it with server truth. δ is a fixed small constant, NOT the ordering tiebreak — round 2 noted that making the tiebreak depend on δ collides rapid sends; that job now belongs to localSeq, leaving δ free to be a simple "just past the last known server time" nudge. The alternative — a pending lane outside the authoritative timeline until a server timestamp places the row — is a B4/UI policy call, flagged below.

The writer census — every path that mutates a messages row

Scope first, because the closure claim is only as good as its boundary (round-3 catch). This census governs writers of the messages table — the table whose invariants (U/A/R/O) carry the app's correctness. Round 1 found one un-audited message-writer (W4); round 2 found two more (W5 retry, W6 delete); round 3 caught that "every path that can touch a row" silently overclaimed — the channels table has its own writers the message-census never covered. So the rule is now precise: every path that mutates a messages row is W1–W6 or it violates the census; the channels table and the message-table's non-mutating paths are enumerated separately below. An un-named message-writer is exactly where the next duplicate hides — but the boundary of the claim is stated, not assumed.

The channels table — its own (small) census

The channels table has no serverUlid/U-equivalent, so it needs no dedup machinery — but round 3 was right that its writers must still be named:

  • CH1 — channel-list sync (GET /v1/channels on launch / refresh): upsert each channel by id (server-authoritative). The sole Phase-1 channel-table writer.
  • CH2 — channel lifecycle (delete / archive / leave) — Phase 2, but named because of its blast radius: when a channel goes away, its messages must NOT be silently mass-deleted by an implicit DELETE CASCADE (round-3 "hidden bomb" — a DB-level writer that bypasses every message invariant). Phase-1 decision: NO foreign-key CASCADE between channels and messages — message cleanup for a defunct channel is an explicit, audited application writer when it lands (it joins the message census then), never an invisible DB trigger.

Proven non-mutating paths (named so they can't masquerade as un-audited writers)

  • migration / onUpgrade — schema-only by contract; adds columns/indexes, performs no row mutation (no backfill rewriting serverUlid, no enum re-keying). A future migration that genuinely must touch rows becomes an audited message-writer and joins the census.
  • read-time / watchChannel mapping — strictly read-only; lenient enum decode lives in the Dart mapping layer, NEVER as a write-back. A read that "repairs" a row is forbidden.
  • connection lifecycle (disconnect / reconnect / offline) — explicitly a NON-writer (round-3): a dropped connection does NOT bulk-transition sending → failed. Pending rows stay sending and re-drain on reconnect (drain-first); a row goes failed ONLY via an explicit ErrorFrame (W4). Naming this forecloses the plausible-but-wrong "mark everything failed on disconnect" implementation, which would strand or double-send messages.
W2 — ack reconcile, the subtle one

ON ack(client_msg_id=c, msg_id=u, created_at=t):

  • If a row R_c exists with clientTempId == c and no other row holds serverUlid == u: UPDATE R_c SET serverUlid=u, createdAt=t, state=sent. (The happy path.)
  • If a row R_u already holds serverUlid == u (history fetched it first — the race): collapse, merging server truth INTO the optimistic row. In one transaction: UPDATE R_c SET serverUlid=u, state=sent AND copy ALL server-authoritative fields from R_u onto R_c — createdAt, sender(senderUserId/senderKind/senderLabel), body, kind, replyToId, channelId — then DELETE R_u (all three statements in the one txn, per Invariant A). The merge direction is server-wins-on-every-field, optimistic-wins-only-on-PK: we keep R_c.clientTempId and its localSeq (UI continuity — a list keyed on clientTempId doesn't see the row swap identities and flicker) but every other value comes from R_u. (Round 2 caught the earlier SQL copying only body/sender/createdAt while the prose claimed "every value" — the field list is now exhaustive so a future divergent field, e.g. a server senderLabel correction, isn't silently kept stale. Phase-1 fields are identical via idempotency, so this is edit-safety insurance, not a live fix.)
  • If no R_c (E1 — app restarted, optimistic row lost but server acked): drop the ack AND schedule a history repair. The ack carries no body/channel_id/sender, so a row can't be materialized from it — but "log and hope history covers it" is not enough (round-2 catch: if the app crashes before the next history fetch, the message is absent indefinitely). So on an orphan ack, enqueue a bounded history-repair for that channel at-or-before msg_id, making recovery an explicit scheduled action rather than an implicit assumption. The repair task must be PERSISTED (round-3: an in-memory schedule followed by a crash leaves the same indefinite-absence hole it exists to close) — it is owned by B4, durable across restart. (Confirmed: envelopes.py ack payload is ids-only.)
W3 — inbound dedup-upsert

A MessageFrame or history MessageView carries msg_id only:

  • If a row already holds serverUlid == msg_id: UPDATE it with the frame's server-authoritative fields — do NOT blind-drop. For Phase-1 immutable messages the fields are identical so the update is a no-op, but the upsert shape (not drop) is what keeps W3 edit-safe: a Phase-2 edit frame for an existing msg_id must land its new content, not vanish. Identity guard (round-2): ULIDs are globally unique, so a matched row with a different channelId is impossible under a correct gateway — treat that mismatch as corruption (log + fail loudly), never silently overwrite channelId/sender identity onto the existing row.
  • Else INSERT (clientTempId=msg_id, serverUlid=msg_id, state=sent).
  • The UNIQUE(serverUlid) index is the backstop: if W3's check races W2's update, the insert throws UNIQUE constraint failed. The catch handler does NOT swallow-and-drop (that would discard the inbound frame's data — cage-match: "data incinerator"). It re-reads the now-present row and applies the same server-field UPDATE, so the frame's payload is never lost.

Reconnect ordering — drain-first (resolves the open question, but does NOT eliminate the race)

The handoff flagged: outbox-drain vs history-fetch ordering on reconnect — drain-after-fetch can duplicate when a history server-id matches an outbox item. Resolution, in two layers:

  1. Policy (B4, minimizes the hazard): drain the outbox BEFORE fetching history. Re-send every serverUlid IS NULL AND state != failed row (same clientTempId → same wire client_msg_id). Each returns an ack (W2) that stamps the ULID. History fetch then becomes pure dedup-by-ULID.
  2. Mechanism (this component, guarantees correctness): W2's collapse-on-collision. Drain-first only reduces the window — acks are async, so a history insert can still land a ULID before its ack arrives. W2's collapse (and the UNIQUE backstop) is what makes the residual race safe, not just unlikely. Do not rely on ordering for correctness; rely on the invariant. Ordering is for clean DeliveryState lifecycle (the user sees sending → sent, not a flicker through a second row).

This split is the component boundary in action: B4 picks the order; the cache guarantees no duplication at rest regardless of whether B4 got the order right — provided (i) invariant A holds (no transient duplicate is observed mid-collapse) and (ii) W1 commits before the wire send, so a drop-after-create reconnect always finds the optimistic row to drain rather than materializing a second one (cage-match: the ack is the only join frame, so a lost optimistic row can't be re-correlated from a later history insert). Both are now writer/invariant preconditions above, not assumptions.

Invariants (stated before the mechanism)

  • U (uniqueness): every NON-NULL serverUlid is UNIQUE — at most one row per ULID, enforced by the index AND by every writer's pre-check. The "non-null" is load-bearing precision (round-2 catch): SQLite's UNIQUE permits multiple NULLs, which is exactly what lets many un-acked optimistic rows coexist; W1 therefore cannot pre-check uniqueness (its serverUlid is null) — only W2/W3, which write a non-null ULID, do. The invariant — duplication, the app's worst failure, is impossible at rest if U holds across W2/W3 (the non-null writers). But U fences the row, not every path that crosses it — see invariants A (stream atomicity) and R (reply join-key), the ways a duplicate or stranded reference leaks around a sound U.
  • A (stream atomicity — U holds in motion, not just at rest): the W2 collapse and every W3 upsert run in a single db.transaction(() async {…}), and watchChannel emits only after that transaction commits. This is a concrete drift contract, not a hope (round-2 catch — "a Carnot engine drawn with friction omitted"): drift coalesces a watcher's re-query to post-commit only when the multiple statements run inside one transaction(); a delete-then-update done as two bare awaited calls forfeits A and lets the watcher observe the intermediate [R_c(null), R_u(u)] — a transient observed duplicate even though the rest-state is one row. Implementation constraint: any multi-statement writer (W2 collapse, W3 conflict-then-update) MUST be wrapped in transaction(); a sequence of bare awaits is a bug, not a style choice. The test plan asserts emissions, not just final row count.
  • R (reply join-key): reply resolution joins on serverUlid, NEVER on clientTempId. A replyToId is always a server ULID (data-layer §B3); resolving it via the PK would strand it when W2 collapse deletes R_u (whose PK was the ULID). Because collapse preserves the ULID on R_c, a serverUlid-join survives it; a PK-join does not. (Cage-match catch — the original draft left the join column unspecified.)
  • O (outbox-as-query): there is no outbox table. The outbox is SELECT * FROM messages WHERE serverUlid IS NULL AND deliveryState != 'failed' ORDER BY createdAt. Send-state lives in exactly one row and cannot drift (chosen over a separate table precisely to avoid the two-table load/write-symmetry hazard).
  • Load/write symmetry: a row with serverUlid == null round-trips load AND save without throwing. Inherited from the data layer's two-id design; the schema makes serverUlid nullable (no sentinel) to preserve it.
  • Reactive lifetime: watchChannel(channelId) returns a Stream<List<Message>> that emits the ordered list on every relevant write and closes when its listener cancels. No leaked controllers (the transport-seam review caught a leak here once — same discipline).
  • Lenient decode on read: senderKind/kind/channel kind decode through the enum fromWire (unknown → safe default), never throw — a forward-compat string in the DB (written by a future client version, then opened by this one after a downgrade) must not brick load.

C1W2 collapse vs. a UI scroll anchor on the optimistic clientTempId. Be honest (round-2 correction): keeping clientTempId keeps the row's IDENTITY stable, but the createdAt swap (clamped client time → server time) IS a guaranteed potential re-sort — the row can change position relative to its neighbors. We have not "prevented flicker," we've narrowed it to a position change without an identity change. For Phase-1 (provisional clock clamped close to real, immutable body) the delta is sub-second and the move is small, but the list test must MEASURE the position delta, not assume it away.

C5two devices / two app instances same account — out of scope Phase 1 (single client), but note it: clientTempId uuids won't collide across devices; ULIDs are server-authoritative, so U still holds. No design debt incurred.
C6a failed row (gateway ErrorFrame for our send, W4) — excluded from the outbox query (O), so it won't auto-redrain. Manual retry is W5 (now a first-class writer in the census): failed → sending, same clientTempId, bumped localSeq, only while serverUlid IS NULL.
C7Server-authoritative edits / deletes / moderation redactions. A msg_id we already hold can return with a corrected body, sender label, deletion tombstone, or moderation state. The original "drop-if-present" would freeze stale local truth forever — lost server state. W3's upsert-on-conflict is the structural shape for field corrections; but the Phase-1 schema has NO tombstone/moderation/edit-version columns, so the cache does NOT structurally prevent lost-delete/moderation state today (round-2 honesty correction — don't overclaim). Deletes route through W6 (tombstone-based, deferred). The per-field merge policy and the tombstone column land with the edit/delete phase; Phase 1 commits only the upsert shape so the door isn't nailed shut.
C8Gross clock skew (cage-match: "temporal wormhole"). Clamped provisional createdAt keeps the row in the live view; localSeq (not δ) orders sibling clamped sends; the ack lands it in server order. Test forces a +years clock AND two rapid sends → no teleport AND the two sends keep compose order.
C9Optimistic reply to a not-yet-acked parent (round-2 catch). Invariant R requires replyToId to be a server ULID — but a user replying to a message whose ack hasn't landed has only the parent's clientTempId. Phase 1 carries replyToId on the wire but the UI ignores it, so this can't arise yet. Precondition for when replies go live: either forbid composing a reply until the parent is acked, or store the parent's clientTempId and rewrite it to the ULID on the parent's W2 ack. Named now so Invariant R isn't silently false the day replies ship.

Test plan (ATDD — written before impl)

  • W1 optimistic insert → outbox query returns it; stream emits it as sending.
  • W2 happy optimistic row + ack(sameTempId) → one row, serverUlid set, state=sent.
  • W2 collapse history inserts ULID u (row R_u) with server body/sender; then ack(c→u) for optimistic R_c → one row remains, clientTempId==c, serverUlid==u, and its body/sender/createdAt are R_u's (server) values, state=sent. (The crux test — asserts merge direction, not just count.)
  • A stream-atomicity drive the W2 collapse while a watchChannel listener records EVERY emission → no emitted list ever contains two rows for one serverUlid (no transient observed duplicate mid-collapse).
  • R reply-join message v with replyToId=u; collapse the optimistic row onto serverUlid=u → v's reply still resolves (join is on serverUlid, survives the R_u delete).
  • W3 upsert insert ULID u; deliver MessageFrame(msg_id=u) with a CHANGED field → still one row, field UPDATED (not dropped). (Phase-1 fields identical → no-op; the test pins the upsert shape for Phase 2.)
  • W3 backstop force the race (insert via W3 while W2 mid-flight, same ULID) → caught, the frame's payload is APPLIED via update, one row, no thrown error escapes the ingest loop.
  • W4 error ErrorFrame(ref=c) on a sending row → failed & leaves the outbox; ErrorFrame(ref=c) on an already-sent row (late error) → NO regression (guard: only when serverUlid IS NULL); ErrorFrame(ref=null) → systemic, fails all pending in channel (or surfaces to B4), never silent-drops.
  • W5 retry a failed row → retry → sending, same clientTempId, localSeq bumped (moves to bottom of pending), re-enters outbox; subsequent ack reconciles via W2.
  • localSeq send-order two optimistic sends under a skewed clock clamp to the SAME createdAt → they render in compose order (localSeq tiebreak), not random uuid order.
  • O rows with serverUlid==null & not failed appear in the outbox query; acked/failed rows don't.
  • C8 clock-skew optimistic row with client clock set +years → stays within the live view (clamped), and on ack lands in correct server order without crossing a different sender's message.
  • Symmetry serverUlid==null row → DB write+read round-trips equal.
  • Migration v1→current open a v1-shaped DB under the current schema → no data loss; additive column loads as its default.
  • Migration version-cycle (round-2 ask) create under vNext schema → downgrade and WRITE a row with vCurrent code → re-upgrade and reopen vNext → assert no crash AND the vNext-only column's value survived the vCurrent write (SQLite column-specific UPDATEs don't clear unmapped columns — pin it).
  • Reactive watchChannel emits ordered list on each write; cancels cleanly (no leaked controller).

Open question for B4 (Component 4 — flagged, NOT resolved here)

W2's collapse keeps R_c's clientTempId. The alternative — keep R_u, delete R_c — is simpler (no field copy) but swaps the UI key. B4 owns this call because it owns the UI-facing reconcile policy; the cache supports either via the same atomic transaction. Recommendation: keep clientTempId (continuity), but let the list-rendering test (C1) settle it.
