Design 03 — Drift cache

The local store B4 writes through — schema, the three writer paths, and the one uniqueness invariant that makes the app's worst failure impossible.

Component scope (Phase 1, text-only): the on-device SQLite store (drift package) that holds the one coherent local truth the UI renders and the repository/reconcile engine (B4, Component 4) mutates. Tables + DAO + reactive per-channel streams. No reconcile orchestration here — that is B4. This component owns storage, atomic operations, and invariant enforcement; B4 owns policy (when to drain, how to order, which DeliveryState transition to apply). Drawing that line is the whole point: if the cache enforces "one row per logical message" at the write boundary, B4 cannot duplicate even with a buggy ordering.

Naming: the package is literally drift (formerly moor) — reactive SQLite with build_runner codegen and Stream queries (.watch()). "Drift cache" = the local store; the homophone is a coincidence we lean into.

Reviewed-as-one because the schema + the three writer paths + the uniqueness invariant are a single bug surface; the table columns alone are trivial.

Source of truth — the gateway wire, re-verified by reading the running code

The schema is shaped by what ws.py actually does on the wire — read, not assumed (aiko_chat_gateway/.../realtime/ws.py:66-89):

  1. Ack precedes fanout, on one connection. _handle_send sends the ack (line 82) then hub.fanouts the message_frame (line 89). The sender, if subscribed, receives both: an ack {client_msg_id → msg_id} first, then a message frame carrying only msg_id (no client_msg_id — message_view omits it).
  2. The server is idempotent on client_msg_id. create_outbound returns (row, created). The ack fires unconditionally (line 82); bus-publish + fanout run only if created (line 84). So an outbox re-send is safe: same msg_id back in the ack, no second echo.
  3. The ack is the only frame carrying both ids. It is the sole join point between the client temp id and the server ULID. Everything reconcile depends on this one frame.

These three facts (verified against the running gateway, per Kelvin's ground-in-environment-first shift) are what the schema and writer paths below are built to satisfy.

The duplicate-row trap (why this component exists)

Message.clientTempId is the durable PK, and for server-origin rows clientTempId == serverUlid (message.dart:141). So the same logical message wears two different PKs across its life:

phaseclientTempIdserverUlid (id)deliveryState
optimistic (just composed)<uuid v4>nullsending
after our own ack<uuid v4><ULID>sent
its own fanout echo, naïvely inserted<ULID><ULID>sent

A naïve insert of the echo (or a history row for a message we sent) creates a second row keyed by the ULID — duplication. The defense is structural, not procedural:

Invariant U (the load-bearing one): serverUlid is UNIQUE. At most one row may ever hold a given ULID. Any write that would introduce a second is either deduped (drop) or collapsed (merge) — never inserted. The UNIQUE index is the write-level backstop that turns a raced dedup-check into a caught error instead of a silent duplicate. (Load/write must agree: the same predicate — "this ULID is not already present" — gates both the check and the constraint.)

Schema

Single messages table (the outbox is a query, not a table — see Invariant O), plus channels.

messages
  clientTempId   TEXT  PRIMARY KEY              -- durable, client uuid OR = serverUlid for inbound
  serverUlid     TEXT  NULL  UNIQUE             -- dedup authority; null until acked. ULID = time-sortable
  channelId      TEXT  NOT NULL  (indexed)
  senderUserId   TEXT  NULL
  senderKind     TEXT  NOT NULL                 -- SenderKind.wire; lenient decode on read
  senderLabel    TEXT  NULL                     -- nullable by wire contract (finding #4)
  kind           TEXT  NOT NULL                 -- MessageKind.wire
  body           TEXT  NOT NULL
  replyToId      TEXT  NULL
  createdAt      INTEGER NOT NULL               -- UTC unix millis (drift maps DateTime); provisional for optimistic
  deliveryState  TEXT  NOT NULL                 -- DeliveryState.wire

channels
  id          TEXT PRIMARY KEY
  name        TEXT NOT NULL
  kind        TEXT NOT NULL                     -- ChannelKind.wire; lenient decode
  aikoChannel TEXT NULL                          -- the mapped aiko channel (gateway sends it)

Ordering key: (createdAt ASC, serverUlid ASC). A ULID sorts lexicographically by time, so once acked it gives authoritative order; an un-acked row sorts by its provisional client clock and settles into place when the ack replaces createdAt. Tiebreak on clientTempId for total order.

The three writer paths

Audit EVERY writer — a schema is a contract only if all writers obey it.

writeroperationinvariant it must uphold
W1composer (optimistic send)INSERT (clientTempId=uuid, serverUlid=null, state=sending, createdAt=now)U trivially (serverUlid null); O
W2ack handlerreconcile the optimistic row → set serverUlid, createdAt, state=sentU: collapse-on-collision
W3inbound (fanout echo + history)dedup-upsert keyed on serverUlidU: drop-if-present
W2 — ack reconcile, the subtle one

ON ack(client_msg_id=c, msg_id=u, created_at=t):

  • If a row R_c exists with clientTempId == c and no other row holds serverUlid == u: UPDATE R_c SET serverUlid=u, createdAt=t, state=sent. (The happy path.)
  • If a row R_u already holds serverUlid == u (history fetched it first — the race): collapse. In one transaction: copy any UI-relevant continuity from R_c, delete R_u, then set R_c.serverUlid=u — keeping the optimistic row's clientTempId so a UI list keyed on clientTempId doesn't see the row swap identities and flicker. Net: one row, server truth, stable key.
  • If no R_c (E1 — app restarted, optimistic row lost but server acked): drop + log. The ack carries no body/channel_id/sender, so a row can't be materialized from it; history re-fetch (W3) recovers the message. (Confirmed: envelopes.py ack payload is ids-only.)
W3 — inbound dedup-upsert

A MessageFrame or history MessageView carries msg_id only:

  • If a row already holds serverUlid == msg_id: drop (our own echo, or a re-fetch, or a re-delivery). Optionally refresh server-authoritative fields; never insert.
  • Else INSERT (clientTempId=msg_id, serverUlid=msg_id, state=sent).
  • The UNIQUE(serverUlid) index is the backstop: if W3's check races W2's update, the insert throws UNIQUE constraint failed instead of duplicating — caught, logged, treated as "already present" (it is).

Reconnect ordering — drain-first (resolves the open question, but does NOT eliminate the race)

The handoff flagged: outbox-drain vs history-fetch ordering on reconnect — drain-after-fetch can duplicate when a history server-id matches an outbox item. Resolution, in two layers:

  1. Policy (B4, minimizes the hazard): drain the outbox BEFORE fetching history. Re-send every serverUlid IS NULL AND state != failed row (same clientTempId → same wire client_msg_id). Each returns an ack (W2) that stamps the ULID. History fetch then becomes pure dedup-by-ULID.
  2. Mechanism (this component, guarantees correctness): W2's collapse-on-collision. Drain-first only reduces the window — acks are async, so a history insert can still land a ULID before its ack arrives. W2's collapse (and the UNIQUE backstop) is what makes the residual race safe, not just unlikely. Do not rely on ordering for correctness; rely on the invariant. Ordering is for clean DeliveryState lifecycle (the user sees sending → sent, not a flicker through a second row).

This split is the component boundary in action: B4 picks the order; the cache guarantees no duplication regardless of whether B4 got the order right.

Invariants (stated before the mechanism)

  • U (uniqueness): serverUlid is UNIQUE; at most one row per ULID, enforced by index AND by every writer's pre-check. The invariant — duplication, the app's worst failure, is impossible if U holds across W1/W2/W3.
  • O (outbox-as-query): there is no outbox table. The outbox is SELECT * FROM messages WHERE serverUlid IS NULL AND deliveryState != 'failed' ORDER BY createdAt. Send-state lives in exactly one row and cannot drift (chosen over a separate table precisely to avoid the two-table load/write-symmetry hazard).
  • Load/write symmetry: a row with serverUlid == null round-trips load AND save without throwing. Inherited from the data layer's two-id design; the schema makes serverUlid nullable (no sentinel) to preserve it.
  • Reactive lifetime: watchChannel(channelId) returns a Stream<List<Message>> that emits the ordered list on every relevant write and closes when its listener cancels. No leaked controllers (the transport-seam review caught a leak here once — same discipline).
  • Lenient decode on read: senderKind/kind/channel kind decode through the enum fromWire (unknown → safe default), never throw — a forward-compat string in the DB (written by a future client version, then opened by this one after a downgrade) must not brick load.

Edge cases the reviewer should attack

case
C1W2 collapse vs. a UI holding the optimistic clientTempId as a scroll anchor — does keeping R_c's clientTempId actually prevent flicker, or does the createdAt change reorder it visibly? (Likely fine — same body, near-same time — but verify with a list test.)
C2history pagination inserts an older batch while a live fanout inserts a newer message — does the ordering key keep them sorted without a full re-sort each insert? (drift re-runs the watch query; confirm the index covers (channelId, createdAt, serverUlid).)
C3the UNIQUE(serverUlid) backstop fires (W3 races W2) — is it caught and swallowed as "already present", or does it bubble and kill the ingest loop? (Must be caught; a thrown constraint error in the frame loop is a socket-killer.)
C4schema migration safety — Phase 1 is v1; the first additive column (Phase 2) must have a drift migration. State the migration strategy now (drift MigrationStrategy, onUpgrade) so a later ALTER doesn't wipe the DB. Load/write must agree across versions, not just within one.
C5two devices / two app instances same account — out of scope Phase 1 (single client), but note it: clientTempId uuids won't collide across devices; ULIDs are server-authoritative, so U still holds. No design debt incurred.
C6a failed row (gateway ErrorFrame for our send) — excluded from the outbox query (O), so it won't auto-redrain. Confirm manual retry re-enters it as sending (clears failed).

Test plan (ATDD — written before impl)

  • W1 optimistic insert → outbox query returns it; stream emits it as sending.
  • W2 happy optimistic row + ack(sameTempId) → one row, serverUlid set, state=sent.
  • W2 collapse history inserts ULID u (row R_u); then ack(c→u) for optimistic R_c → one row remains, clientTempId==c, serverUlid==u, state=sent. (The crux test.)
  • W3 dedup insert ULID u; insert MessageFrame(msg_id=u) again → still one row.
  • W3 backstop force the race (insert via W3 while W2 mid-flight, same ULID) → caught, one row, no thrown error escapes the ingest loop.
  • O rows with serverUlid==null & not failed appear in the outbox query; acked/failed rows don't.
  • Symmetry serverUlid==null row → DB write+read round-trips equal.
  • Migration open a v1 DB with the v-current schema → no data loss (once Phase 2 adds a column).
  • Reactive watchChannel emits ordered list on each write; cancels cleanly (no leaked controller).

Open question for B4 (Component 4 — flagged, NOT resolved here)

W2's collapse keeps R_c's clientTempId. The alternative — keep R_u, delete R_c — is simpler (no field copy) but swaps the UI key. B4 owns this call because it owns the UI-facing reconcile policy; the cache supports either via the same atomic transaction. Recommendation: keep clientTempId (continuity), but let the list-rendering test (C1) settle it.
