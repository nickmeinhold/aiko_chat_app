aiko chat · the transport seam

A five-scene explainer

One helm, any engine

The app never touches an engine, a wire, or a socket. It steers one helm — and whatever powers the boat behind the seam can be swapped without the crew ever relearning the wheel.

Scroll to explore
Scene 01

One helm the whole crew steers

The idea

Imagine a boat where the crew never opens the engine hatch. They give one wheel a turn and the boat answers — hail an island, send a word, listen for a reply. Nobody up top needs to know whether a diesel or a sail is doing the work below. There is exactly one set of controls, and every order the app gives, it gives through that.

The single control surface: the app depends only on the ChatTransport interface — connect, subscribe, sendMessage, and streams of messages, acks and errors. It never imports a socket or an HTTP client directly, so the whole app steers one wheel and nothing else.
Scene 02

Swap the engine, keep the wheel

The idea

Behind the helm is a bulkhead — a clean seam — and behind that, the engine room. Today a real engine is bolted in: it dials a harbour over the radio and the post road. Tomorrow you could drop in a different engine, or a dummy one on a test bench, and the crew would never feel the change. The wheel’s shape is a promise; anything that keeps the promise can sit behind the seam.

Interchangeable behind the seam: concrete engines like ChatRestApi and the gateway transport implement the same interface and are swapped in behind it — the live one speaks /v1/* REST and a WebSocket to reach an island through its gateway. Tests bolt in a FakeChatTransport instead. A contract change touches the engines, never the crew.
Scene 03

The papers live in a locked strongbox

The idea

Every crossing needs papers — the sealed letters that prove the boat is allowed into a harbour. You don’t leave those on the deck. They ride below, in a locked strongbox that only the boat can open, and there is exactly one of them. When the radio operator or the helm needs to show papers, they draw from that one box, never a loose copy in a pocket.

One keeper of credentials: auth tokens live in SecureTokenStore, a flutter_secure_storage vault with read / write / clear. It is the single source of truth: both the REST layer and the WebSocket connect read their token from the same box, so there is never a stale copy drifting somewhere else.
Scene 04

When the papers expire, one runner is sent

The idea

Papers go stale on a long voyage. The moment they do, both the radio operator and the helm want fresh ones at once — and if each sent their own runner ashore, the harbour would issue two sets and get confused. So the boat sends exactly one runner. Everyone who’s waiting queues behind that single errand and takes the same fresh papers when the runner returns.

Single-flight refresh: a small TokenProvider seam hands out the access token and, when it’s stale, refreshes it exactly once — the in-flight future is shared so a burst of expiries triggers one round trip, not many. Fresh papers are written back to the one strongbox; if the refresh itself is refused, the boat lands in unauthenticated and heads for the login shore.
Scene 05

The radio’s static never says who you are

The idea

When the radio link drops, the crackle it leaves behind tells you almost nothing — a storm and a slammed door sound the same over the air. So the boat doesn’t try to read the static for “were we turned away?”. A dropped link just means “raise them again.” To learn whether the papers themselves went bad, the boat asks the harbourmaster plainly, over the post road, where the answer is clear.

Auth is decided over REST, not the socket: a WebSocket close code is unreadable or masked across platforms, so a drop is treated as an unclassified reconnect-needed event. The clean signal is a 401 on a REST call, which drives the single refresh; a dead refresh means unauthenticated. Reconnects use capped backoff and re-send the subscribe set — bounded, never a loop.

The whole thing, in one breath

One wheel to steer, one box for the papers, one seam that hides the rest.

The app turns a single helm and reaches any island. Behind the seam, the engine can change, be faked, or be rebuilt, and the crew never notices. The papers stay in one locked box; when they lapse, one runner fetches more; and the boat asks a clear question over the post road rather than reading static off the air.

The seam is deliberately thin: it holds a live connection and a set of papers, and nothing more. Where the words that cross it come to rest — the logbook, the order they’re kept in, the mailbox that waits — is the next boat over: the islands, and the harbours that hold the record.
