Design 05 — Communities (nested servers) + the discovery directory

Introduce a two-level hierarchy — a Community (a "server") owns Channels — and build the discovery directory at the community level. This replaces the current flat channel model. The whole design is shaped so it ships incrementally and never bricks launch: the first phase is an invisible model migration, the directory lands later.

Status: design, not yet built. Decision: nested communities ("Option B"), chosen by Nick 2026-06-29 over the flat-channels-with-tags alternative ("Option A"). Task #6.
Legend: GATEWAY peer-owned (aiko_chat_gateway) · APP this repo. The bulk of v1 is gateway data-model + endpoint work; the app side is a rendering + navigation layer that cannot be built until the contract lands (same seam discipline as auth #38).

1. The shape — and why the migration is the load-bearing part

Today the model is flat: GET /v1/channels returns the channels you're already in, each carrying only {id, name, kind, aiko_channel}. There is no "server", no description, no member count, no discovery endpoint, no join. (See §9 for the full gap list.)

Option B nests that: a Community owns channels, membership is two-level (you join a community, which grants access to its channels), and the directory browses communities. The risk Option A was meant to avoid — that this is a months-long epic blocking launch — is neutralised by one move:

The default-community migration. Every existing channel is assigned to a single seeded community (call it "Aiko"). Every existing user is auto-joined to it. Phase B1 is therefore invisible — channels still render exactly as today, they just now belong to a community under the hood. No user loses a channel; no UI changes; launch is unaffected. The hierarchy becomes visible only in B2 when the directory + community navigation ship. This is how "B now" stops meaning "B blocks everything."

2. Domain model

2.1 The new Community type APP

/// A community ("server") — the top-level grouping that owns channels and
/// membership. The discovery directory ranks and filters these, not channels.
class Community {
  final String id;
  final String name;
  final String? description;
  final String? iconUrl;
  final CommunityVisibility visibility; // public | unlisted | private
  final int memberCount;
  final String ownerId;
  final CommunityCategory category;     // closed set — drives directory grouping
  final String defaultChannelId;        // where a new member lands
  final DateTime? lastActivityAt;       // coarse signal for ranking
}

enum CommunityVisibility { public, unlisted, private; /* fromWire fail-closed */ }
enum CommunityCategory   { general, tech, gaming, art, music, education, social, other; }

Closed sets are enums with fail-closed fromWire (an unknown future visibility/category is dropped or coerced to the safe default), matching the AuthProviderKind / ChannelKind pattern already in the codebase — never stringly-typed.

2.2 Channel gains a parent APP GATEWAY

class Channel {
  final String id;
  final String name;
  final ChannelKind kind;
  final String? aikoChannel;
  final String? communityId;   // NEW — the owning community. Null only for DMs.
}

Wire (GET /v1/channels and the nested forms) adds community_id. Decoding stays lenient (absent ⇒ null), so a pre-migration gateway response still parses — important for the B1 rollout ordering.

IDs are globally namespaced from day one — the one cheap principle adopted from Design 06 §8. Community, channel, and user IDs carry their origin gateway (e.g. id@homeGateway) even while there is exactly one gateway. This costs almost nothing now — it's just "don't backslide into a flat, gateway-bound ID space" — and it is the single discipline that keeps the federation future (06) open without a from-hell migration later. The rest of 06 is deferred; this much is load-bearing for B1 and belongs in the model the moment community_id ships.

2.3 Two-level membership

LevelMeaningv1 rule
Community membership"I belong to this server"Joining a community subscribes you to all its public channels (simple). Per-channel opt-in is a B3 concern.
Channel subscriptionThe WSS subscribe set (already exists)Derived: union of channels across my joined communities. The reconcile engine is unchanged below the derivation.

Roles/permissions are explicitly out of v1 — owner + member only. A role system (Discord-style) is a whole subsystem; deferring it is what keeps B1/B2 tractable. See §8.

3. Discovery operates at the community level

This is the clean payoff of B: "server discovery" is community-level discovery, exactly like Discord's directory. The three sub-problems from the topology framing map cleanly:

Sub-problemv1 answerSignal source
Taxonomy (grouping)CommunityCategory — a closed enum facetNew field on the community
Relevance (ranking)Default: lastActivityAt recency + memberCount. Cheap proxies, no engagement model.Coarse fields, no aggregation pipeline
Visibility (who sees what)Only public communities listed; per-viewer block/takedown predicate applied. See §7.Reuses moderation predicate
Cold-start is real. A brand-new directory has no engagement signal, so v1 ranking deliberately leans on recency + size. Not personalized in v1 — the trigger to add personalization is "rank by overlap with the viewer's existing community memberships." Trending (join-rate / message-rate deltas) and featured (admin-curated) are B3: each needs a new gateway aggregation, so naming them as later keeps v1 shippable.

5. App architecture APP

  • Domain: Community + CommunityMembership; Channel.communityId.
  • Data: discoverCommunities(), getCommunity(), joinCommunity(), listMyCommunities() on ChatRestApi (mirrors the existing method-per-endpoint style).
  • Providers: myCommunitiesProvider (replaces the role of channelsProvider as the subscription root), communityChannelsProvider(id), discoverCommunitiesProvider(query).
  • Reconcile: the subscribed-channel set becomes a derivation — union of channels across joined communities — fed into the unchanged B4 engine. The transport/cache primitives don't move; only where the channel-id list comes from changes.
  • Navigation: a community rail (left) → channel list (within) → chat. On mobile this is a two-level push; the current single-pane ChatScreen grows a parent layer rather than being rewritten.

Ordering invariant: B1 gateway serialization of community_id is safe to ship before the app reads it (lenient decode tolerates its absence and its presence). The default-community seed + auto-join must be a single migration so no channel is ever orphaned (a channel with a null community_id that isn't a DM is a migration bug, not a valid state — assert it server-side).

7. Visibility & concurrent-shrink (the teeth)

The directory is a fourth read path into the same visible-set as the reconnect fence, history paging, and fanout. Per the project's distributed-time invariant (concept_visibility_consistency_distributed_time), it must apply the identical visibility predicate the moderation work (#7) established — not-blocked + not-soft-deleted + not-private-to-this-viewer — now lifted to the community grain.

Design-time question, not a cage-match catch: what shrinks the discover set concurrently? A community flipping public → private, the owner being blocked, a takedown — all between a viewer's discover list and their join. Therefore join fails closed: if the community is no longer visible to this viewer at join time, the gateway rejects (403/404) and the app shows "no longer available" rather than joining something that just went private. The directory listing is a snapshot; only join is the authoritative gate.
  • Three visibilities: public (listed + joinable), unlisted (joinable by direct link, not listed), private (invite/member-only, not listed). Only public enters discover.
  • Client defense-in-depth: the app filters owner-blocked communities client-side too (mirroring the existing blockedUserIdsProvider message filter), so a block takes effect on the next frame without a reload — but the backend predicate is the source of truth.

9. Open questions for the gateway GATEWAY

  • Does the gateway model already have any community/grouping scaffolding (as it already had a latent is_private)? If so, B1 is partly done.
  • Is aiko_channel (the bus mapping) per-channel or could it become per-community? Affects whether the bus topology mirrors the new hierarchy.
  • Ranking source for last_activity_at — is there a cheap "last message ULID per channel" already (the reconnect fence suggests yes) that rolls up to a community?
  • Moderation grain: can a community be taken down as a unit, or only individual channels/messages? The visibility predicate (§7) needs a community-level takedown flag.
