Runbook — Phase-1 human e2e

Driving the Flutter/macOS app by hand against a live gateway, and confirming the Phase-1 wiring holds across the deploy boundary.

Scope. Phase 1, text-only. This is the human e2e — a person clicking the real signed/sandboxed macOS app against a real gateway. It is the only instrument that crosses the packaging + deploy boundary (concept_packaging_boundary_blind_spot): the test pyramid is structurally blind to entitlement, keychain, and deploy-wiring failures. Everything below is grounded in the current source — paths cite real files.

TL;DR — the one command

flutter run -d macos \
  --dart-define=GATEWAY_BASE_URL=https://chat.imagineering.cc

Then: Register a fresh account (a brand-new prod gateway has no users — register, don't log in), send a message, watch it go sending→sent, quit the app, relaunch, confirm the message survived. That round trip exercises the whole Phase-1 spine.

The same binary targets localhost with no flag (GatewayConfig.fromEnvironment() defaults to http://localhost:8095 — lib/app/config.dart:22). wsBaseUrl is derived from the HTTP base (https→wss), so you only ever set the one define and the two can't drift.

Pre-flight — the macOS gotchas already paid for

These two bugs were the test pyramid's blind spots; both are fixed in main. They are listed so a failed launch is diagnosed in seconds, not rediscovered.

1 — Network access requires the network.client entitlement (fixed, PR#13)

A sandboxed macOS app cannot reach the gateway without com.apple.security.network.client in the entitlements. If a fresh build ever can't connect and localhost works from curl, check the entitlements first (macos/Runner/*.entitlements). The symptom is a silent connection failure, not an error dialog.

2 — Keychain -34018 on token storage (fixed, PR#15)

Secure token storage failed with errSecMissingEntitlement (-34018) until the app was signed with the Enspyr dev team + keychain-access-groups. If login succeeds but the session doesn't persist across relaunch (you're kicked to /login every cold start), suspect signing/keychain, not the auth logic — the token never got written.

Launch

  1. Confirm the gateway is live: curl -sS -m8 https://chat.imagineering.cc/v1/healthz returns 200. (Adjust the health path to whatever incr 3 exposes.)
  2. From the app repo root, run the one command above.
  3. The app boots to /splash while cold-start session restore runs (auth_controller.dart:60 _restoreSession). With no stored tokens it falls through to /login — expected on a clean machine.

Verify checklist — what each click proves

Each row is a Phase-1 invariant made observable. The point of the human e2e is that these are the exact guarantees the data layer was built to provide — so a green run here is the design working end to end, not a smoke test.

Do thisExpectProves
Register a new account (username + display name + password; toggle in login_screen.dart). Lands directly on the chat screen — register signs straight in. POST /v1/auth/register returns tokens; setTokens → router guard trips logged-in → / (auth_controller.dart:87, router.dart:50).
Send a message. Row appears immediately with a single "sending/sent" checkmark, before any network round trip. B-optimistic — the optimistic row commits to the cache before the wire send, so an app kill between the two loses nothing (design 04).
Watch the row after send. The indicator settles to sent within a beat. The ack stream reconciled the temp row to its server ULID (_onAck / reconcileAck). Phase 1 collapses sent/delivered/read into one tick by design — task #1272.
Quit the app (⌘Q) and relaunch with the same command. You land straight on chat (no login), and your message is still there. Session survives via me()-backed restore + the per-user file-backed Drift cache survives restart. This is the deploy-boundary proof — code-supports ≠ deployed-truth.
Log out, then log back in. Clean transition; no stale messages from the previous session leak in. autoDispose session scope tears the cache + socket down leaf-to-root (Carnot C1/C3) — session isolation falls out of the lifecycle.
Pull the network briefly (toggle Wi-Fi), then restore it. You are not logged out; the socket reconnects and catches up. The transient-vs-terminal 401 boundary: a blip is disconnected (retried), only a real token rejection is unauthenticated (logout) — auth_error_boundary, the whole point of the seam.

Why by hand, and not just the test pyramid

concept_packaging_boundary_blind_spot — a live signed/sandboxed run is the only instrument that crosses the packaging boundary. Both macOS bugs above (entitlement, keychain) passed every unit/integration test and still broke the real app, because the harness never signs, sandboxes, or deploys. code-supports ≠ deploy-wired ≠ verified-live: green CI proves the code, this runbook proves the deployed truth.
